## Workflow

### Step 1: Get the Changes

If the user hasn't provided a diff, run one of the following to collect the changes:

```bash
# Staged changes (most common)
git diff --staged
 
# Unstaged changes
git diff
 
# Both staged and unstaged
git diff HEAD
 
# Last commit
git show HEAD
```

If the user provides the diff directly in the message, use that instead.

---

### Step 2: Analyze the Diff

Before writing anything, read the diff carefully and extract:

1. **What files changed** — group by feature/module/layer
2. **What was added, removed, or modified** — be specific
3. **Why the change was likely made** — infer from context, naming, and structure
4. **What the impact is** — does it affect performance, security, UX, API contracts, DB schema, dependencies, etc.?

---

### Step 3: Write the Commit Message

Use the following structure:

```
<type>(<scope>): <short imperative summary> (max 72 chars)
 
## What changed
- <specific change 1>
- <specific change 2>
- <specific change 3>
(as many as needed — be concrete, not vague)
 
## Impact on the project
- <effect on behavior, users, performance, etc.>
- <breaking changes if any>
- <dependencies affected>
```

---

### Commit Types

Use conventional commit types:

| Type | When to use |
|------|------------|
| `feat` | New feature or capability |
| `fix` | Bug fix |
| `refactor` | Code restructure without behavior change |
| `perf` | Performance improvement |
| `style` | Formatting, linting, no logic change |
| `test` | Adding or updating tests |
| `docs` | Documentation only |
| `chore` | Build config, dependencies, tooling |
| `ci` | CI/CD pipeline changes |
| `security` | Security-related fix or hardening |

---

### Rules — Non-Negotiable

1. **Never mention AI, Claude, GPT, LLM, "generated by", "assisted by", or any automation tool** — the commit must read as if a senior developer wrote it manually.
2. **Use imperative mood** — "Add validation" not "Added validation"
3. **Be specific** — name the actual functions, files, modules, or config keys that changed
4. **No filler phrases** — avoid "various improvements", "minor updates", "some changes"
5. **Scope = the module/package/layer affected** — e.g., `auth`, `api`, `db`, `ui`, `config`
6. **If the diff is large**, group related changes and summarize each group — don't list every single line
7. **Impact section must be honest** — if there's a breaking change, say so explicitly

---

### Output Format

Always output:

1. The full commit message in a code block (ready to copy-paste)
2. A brief explanation (2-3 sentences) of what you inferred and why you chose that structure — **outside** the code block, in plain language

If you're uncertain about the intent of a change, say so briefly and offer 2 commit message variants.

### Example Output

**Input diff summary**: Added JWT expiry check to auth middleware, removed old session cookie logic.

**Output**:

```
fix(auth): enforce JWT expiry validation and remove legacy session logic
 
## What changed
- Added expiry timestamp check in `validateToken()` before granting access
- Removed `session_cookie` parsing from `AuthMiddleware` (deprecated since v2.1)
- Updated `UnauthorizedError` to include expiry reason in response body
- Added unit tests covering expired, malformed, and missing token scenarios
 
## Impact on the project
- Clients with expired JWTs will now receive 401 with explicit expiry message
- Legacy session-based clients will break — coordinate with frontend before deploying
- Reduces attack surface by eliminating dual-auth code path
```

---

### Edge Cases

- **Empty diff**: Ask the user to run `git diff --staged` and share the output
- **Binary files only**: Note the files changed and ask for context about what changed and why
- **Massive diff (500+ lines)**: Focus on the highest-impact changes, group the rest, and note that the commit covers a large changeset
- **Multiple unrelated changes**: Suggest splitting into separate commits and offer a message for each

Return ONLY the commit message in the conventional format, nothing else.
